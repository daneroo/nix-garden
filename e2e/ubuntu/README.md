# Ubuntu 24.04 + xfce4/xrdp on Proxmox

Fully automated provisioning of an Ubuntu 24.04 VM with xfce4 desktop and xrdp.

## Usage

```bash
./provision.sh           # defaults: cloud image, VM 996
./provision.sh --help    # all options
```

Runs from your Mac, SSHes into Proxmox (`hilbert`), creates the VM, installs the OS, then installs the desktop over SSH.

- **Cloud image** (default) -- ~3.5 min total
- **ISO installer** (`--mode iso`) -- ~6 min total

Connect: `open ubuntu/connect-vm996-<ip>.rdp` (generated by provision.sh)

Destroy: `ssh root@hilbert 'qm stop 996 && qm destroy 996 --purge'`

## How it works

### Cloud image mode (default, ~3.5 min)

Uses Ubuntu's pre-built cloud image (`noble-server-cloudimg-amd64.img`) — a qcow2 disk with the OS already installed. No installer runs at all.

- **Import disk** -- `qm set --scsi0 local-zfs:0,import-from=<path>` imports the cloud image as the VM's boot disk and resizes it to 64G
- **Cloud-init via Proxmox** -- `qm set --ide2 local-zfs:cloudinit` tells Proxmox to **automatically generate a seed ISO** (labeled `cidata`) containing `user-data`, `meta-data`, and `network-config`. This ISO is built from the settings passed via `--ciuser`, `--sshkeys`, `--ipconfig0` and attached as a virtual CD-ROM. The VM's cloud-init agent reads it on first boot. No manual seed ISO or snippet file needed
- **No password via cloud-init** -- Proxmox's built-in cloud-init only supports plaintext `--cipassword`, which we avoid. Instead, the password is set post-boot over SSH using `chpasswd -e` with a SHA-512 hash
- **Why not `--cicustom`?** -- Proxmox supports `--cicustom user=local:snippets/file.yaml` to provide a full cloud-init config (with packages, runcmd, hashed passwords, etc.). But a custom `user` snippet **completely replaces** the auto-generated user-data — you lose `--ciuser`, `--sshkeys`, `--ipconfig0` and must duplicate everything in the snippet. Not worth the complexity when we can just install packages post-boot via SSH
- **Serial console** -- `--serial0 socket --vga serial0` gives a text console (accessible via `qm terminal 996`) instead of a virtual GPU. Fine for RDP — xrdp renders in software. noVNC in Proxmox UI shows serial text only
- **IP resolution** -- cloud images don't have qemu-guest-agent pre-installed, so we resolve the VM's IP via ping sweep + ARP table lookup using the VM's MAC address
- **Wait for cloud-init** -- `cloud-init status --wait` blocks until cloud-init finishes, which holds the apt lock. Desktop packages are installed only after this completes
- **Desktop install over SSH** -- xfce4 + xrdp installed via `apt-get` after the OS is up

### ISO installer mode (fallback, ~6 min)

Uses the stock Ubuntu server ISO (`ubuntu-24.04.4-live-server-amd64.iso`) with fully automated autoinstall.

#### The autoinstall problem

Ubuntu 24.04's subiquity installer shows a "Continue with autoinstall?" confirmation prompt unless the word `autoinstall` appears on the kernel command line. You can't modify the ISO's GRUB config without `xorriso`. A previous approach used `qm sendkey` to blindly type "yes" — fragile and unreliable.

#### The solution: direct kernel boot

We bypass GRUB entirely:

- **Extract kernel/initrd** -- mount the stock ISO, copy `casper/vmlinuz` and `casper/initrd` to `/tmp`
- **Direct QEMU boot** -- create the VM with `--args '-kernel /tmp/ubuntu-vmlinuz -initrd /tmp/ubuntu-initrd -append "autoinstall ds=nocloud;"'`
  - `autoinstall` -- suppresses the confirmation prompt
  - `ds=nocloud;` -- tells cloud-init to find config on a local disk (our seed ISO)
- **Seed ISO** -- a tiny ISO (labeled `cidata`) containing the autoinstall config (`on-proxmox-iso-seed.yaml`), built on Proxmox via `genisoimage`
- **Shutdown on complete** -- `shutdown: poweroff` in the seed config makes the installer power off the VM instead of rebooting (a reboot would loop back into the installer since `--args` persists)
- **Reconfigure and restart** -- script detects `stopped` status, removes `--args`, detaches both CDROMs (ide0/ide2), sets boot to disk-only, then starts the VM
- **Desktop install over SSH** -- xfce4 + xrdp are installed via `apt-get` after the OS is up (not during autoinstall, where apt failures are fatal and hard to debug)

#### What didn't work (ISO mode)

- **`qm sendkey` loop** -- blindly typing "yes" to the confirmation prompt. Timing-dependent, broke silently
- **Installing xfce4 during autoinstall** -- `packages:` in seed config runs apt inside the installer environment. If it fails (network issues, dependency problems), the entire install fails with no easy way to debug
- **Waiting for guest-agent during install** -- the agent runs in the installed OS, not the installer. It never responds until the OS boots from disk
- **Letting the installer reboot** -- with `--args` set, QEMU always uses direct kernel boot, so it re-enters the installer in a loop
- **Deleting seed ISO before detaching** -- the EXIT trap deleted the seed ISO file while it was still attached as ide0, causing `qm start` to fail with "volume does not exist"

## Files

- `provision.sh` -- runs on Mac. Orchestrator: parses CLI, provisions VM, waits for SSH, installs desktop
- `on-proxmox-common.sh` -- shared constants (VM sizing, paths, SSH key) and helpers (start_vm, ping sweep)
- `on-proxmox-img.sh` -- runs on Proxmox. Cloud image: import disk, configure cloud-init, boot
- `on-proxmox-iso.sh` -- runs on Proxmox. ISO: extract kernel, direct boot, wait for install, reconfigure for disk boot
- `on-proxmox-iso-seed.yaml` -- cloud-init autoinstall template, expanded via `envsubst` from common.sh values (ISO mode only)
